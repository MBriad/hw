# 架构合规性分析报告

评估当前代码架构是否符合 `skill/clean-arch-app/SKILL.md` 中定义的 Clean Architecture 规范，以及用户是否可以安全地修改 UI。

## 逐条合规检查

### ✅ 通过项

| 规则 | 状态 | 说明 |
|------|------|------|
| domain/ 零 @kit.* 依赖 | ✅ 完全合规 | 所有 domain 文件无框架依赖 |
| usecases/ 仅依赖 ports 接口 | ✅ 完全合规 | 3 个 usecase 都只导入 domain/ports |
| viewmodels/ 仅依赖 ports 接口 | ✅ 完全合规 | HeatmapVM 不导入任何 usecase 或 adapter |
| 子组件遵循 Humble View | ✅ 完全合规 | DayCell、HeatmapGrid 等只渲染 + 发出回调 |
| @Provide/@Consume 主题分发 | ✅ 完全合规 | tokens 和 themeName 正确分发 |
| stateStyles({ pressed }) | ✅ 完全合规 | 所有可交互元素都有按压状态 |
| Spacing/Radius tokens | ✅ 完全合规 | 无硬编码数字 |
| 测试 mock 用命名类 | ✅ 完全合规 | 无 object literal 实现 interface |

### ❌ 违规项

| 规则 | 违规位置 | 严重度 | 说明 |
|------|---------|--------|------|
| **依赖方向：pages → viewmodels only** | `HeatmapPage.ets` 第 6-11 行 | 🔴 严重 | 直接导入 4 个 adapter 和 2 个 usecase |
| **Composition Root = EntryAbility** | `HeatmapPage.ets` 第 36-43 行 | 🔴 严重 | HeatmapPage 的 `aboutToAppear()` 创建所有实例，EntryAbility 导入被注释 |

| **@Provide/@Consume 仅用于框架状态** | `HeatmapPage.ets` | 🟡 中等 | 用 @Provide 传 tokens 是合规的，但 themeName 作为业务状态也通过 @Provide 传，边界模糊 |
| **测试隔离** | `HeatmapVM.test.ets` | 🟡 中等 | 使用了具体 `SkinProviderImpl` 而非 mock |

## 能否改 UI？—— 可行性分析

### 结论：**可以安全修改 UI，但有一个前提**

#### ✅ 你可以安全修改的部分

所有 `pages/components/` 下的子组件完全合规，修改它们**不会**破坏架构：

| 组件 | 改什么 | 风险 |
|------|--------|------|
| **DayCell** | 外观、动画、布局、颜色 | 零风险 — 纯展示组件 |
| **HeatmapGrid** | 网格布局、间距、行列数 | 零风险 — 纯展示组件 |
| **CheckInInput** | 输入框样式、按钮外观 | 零风险 — 通过回调发出意图 |
| **SkinSwitcher** | 皮肤选择器布局/样式 | 零风险 — 通过 onSwitch 回调 |
| **QuoteBanner** | 引语展示样式/动画 | 零风险 — 纯展示 |
| **ParticleOverlay** | 粒子效果/动画 | 零风险 — 纯展示 |
| **EasterEggTester** | 测试面板样式 | 零风险 — 通过回调发出意图 |

修改 `common/` 下的设计 token（SpacingTokens、RadiusTokens、GlassTheme）也是安全的。

#### ⚠️ 需要注意的部分

| 修改目标 | 风险 | 建议 |
|---------|------|------|
| **HeatmapPage.ets 的布局** | 中风险 | 页面内联了组合根逻辑，改 UI 布局时别碰 `aboutToAppear()` 里的依赖创建 |
| **主题持久化** | 中风险 | HeatmapPage 和 ThemeProvider 有重复逻辑，改一处要同步另一处 |
| **新增组件** | 低风险 | 新组件只要遵循 @Consume tokens + 回调模式即可 |
| **新增页面** | 高风险 | 新页面也需要依赖注入，当前没有共享的组合根机制 |

#### 🔴 不建议动的部分（除非重构）

`HeatmapPage.ets` 的第 6-13 行（导入）和第 36-43 行（依赖组装）—— 这是架构违规的根源，但改它属于重构而非 UI 修改。

### 依赖方向图（当前实际）

```
                    ┌─────────────────────┐
                    │   HeatmapPage.ets    │
                    │  ❌ 直接导入 adapters │
                    │  ❌ 直接导入 usecases │
                    │  ❌ 直接导入 @kit.*   │
                    └──┬──────┬──────┬─────┘
                       │      │      │
              ┌────────┘      │      └────────┐
              ↓               ↓               ↓
     ┌────────────┐   ┌────────────┐   ┌────────────┐
     │  adapters  │   │  usecases  │   │  viewmodels│
     │(@kit.ArkData)│  │            │   │  HeatmapVM │
     └─────┬──────┘   └─────┬──────┘   └──────┬─────┘
           │                │                  │
           │                ↓                  │
           │         ┌────────────┐            │
           └────────→│   domain   │←───────────┘
                     │   + ports  │
                     └────────────┘
```

问题：`HeatmapPage` 跨过了 viewmodel 层直接依赖 usecases 和 adapters，违反了 `pages → viewmodels → usecases → domain` 的单向依赖规则。

## 如果要修改 UI 的具体建议

### 1. 纯样式修改（零架构风险）
- 修改 DesignTokens 中的颜色值（[domain/SkinTheme.ets](entry/src/main/ets/domain/SkinTheme.ets)）
- 修改 Spacing/Radius tokens（[common/SpacingTokens.ets](entry/src/main/ets/common/SpacingTokens.ets)、[common/RadiusTokens.ets](entry/src/main/ets/common/RadiusTokens.ets)）
- 修改任意组件的 `build()` 方法内的布局和样式

### 2. 组件重构（低架构风险）
- 拆分 HeatmapPage 的内联 header 为独立组件
- 新增组件（遵循 @Consume tokens + 回调模式）
- 替换 ForEach 为 LazyForEach（如果性能需要）

### 3. 布局重排（低架构风险）
- 修改 HeatmapPage 的 `build()` 中的组件排列
- 注意不要修改 `aboutToAppear()` 中的依赖创建代码

### 4. 不建议在改 UI 时同时做的事
- 不要把 HeatmapPage 的组合根逻辑搬到 EntryAbility（这是重构，不是 UI 修改）
- 不要在组件中直接导入 adapters 或 usecases
- 不要在 build() 中添加业务逻辑

## 验证方式

修改 UI 后应执行：
1. `arkts_check` 检查所有修改的 .ets 文件
2. `assembleHap` 编译验证
3. `test` 运行单元测试
4. 在模拟器/设备上实际查看 UI 效果（4 个主题都要验证）

## 回滚方案

UI 修改一般只涉及 .ets 文件，可通过 git revert 回滚。不涉及 module.json5 或资源配置变更时回滚简单。
