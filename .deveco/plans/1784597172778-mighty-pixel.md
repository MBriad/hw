# Plan: Clean Architecture Audit & Navigation Page Component Extraction

## Goal

Audit the codebase against the `clean-arch-app` SKILL.md and fix all violations. Extract the 4 navigation page content areas (`dataPage`, `logsPage`, `corePage`, `userPage`) currently inlined as `@Builder` methods inside `HeatmapPage.ets` into proper `@Component` structs, each in their own file under `pages/components/`.

## Scope / Non-goals

**In scope:**
- Fix all Clean Architecture violations identified in the audit
- Extract 4 navigation page sections into dedicated components
- Remove dead/unused code
- Fix ArkTS strict-mode type issues

**Out of scope:**
- New features or UI redesign
- Changing the navigation mechanism (HorizontalNavigation stays as-is)
- Adding new tests beyond what's needed for refactoring
- Modifying usecase/domain logic

### Navigation page content — not yet componentized:

The 4 tab pages (`dataPage`, `logsPage`, `corePage`, `userPage`) are all `@Builder` methods inside `HeatmapPage.ets` with inline UI code ranging from 40-100 lines each. Per Clean Architecture, these should be extracted to dedicated `@Component` structs as Humble Views.

## Design

### Step 1: Convert domain interfaces to classes (ArkTS strict mode)

The `EasterEgg`, `CheckIn`, `StreakInfo`, `HeatmapCell` interfaces must become `class` definitions. The `ThemePalette` record must use `new DesignTokens(...)` instead of object literals with `as DesignTokens`.

**Files affected:**
- `domain/EasterEgg.ets` — `EasterEgg` interface → class
- `domain/CheckIn.ets` — `CheckIn` interface → class
- `domain/Streak.ets` — `StreakInfo` interface → class, `HeatmapCell` interface → class
- `domain/SkinTheme.ets` — Add constructor to `DesignTokens`, convert `ThemePalette` to use `new DesignTokens(...)` for each theme
- `usecases/RecordCheckIn.ets` — Change object literal to `new CheckIn(...)`

### Step 2: Fix DIP violation — move theme persistence to adapter

Theme persistence (reading/writing `@kit.ArkData.preferences`) is currently in both `HeatmapPage.ets` and `ThemeProvider.ets`. Create a `ThemeRepository` port in `domain/ports/` and a `PreferencesThemeRepo` adapter.

**New files:**
- `domain/ports/ThemeRepository.ets` — interface with `load(): Promise<SkinTheme>` and `save(theme: SkinTheme): Promise<void>`
- `adapters/PreferencesThemeRepo.ets` — implements port using `@kit.ArkData`

**Files modified:**
- `HeatmapPage.ets` — remove `loadPersistedTheme()`/`persistTheme()`, remove `import { preferences } from '@kit.ArkData'`
- `ThemeProvider.ets` — same removal
- `EntryAbility.ets` — wire `PreferencesThemeRepo`, inject into page or VM

### Step 3: Extract 4 navigation page components

Extract each `@Builder` method from `HeatmapPage.ets` into its own `@Component` file:

| Builder Method | New Component File | Key Props |
|----------------|-------------------|-----------|
| `dataPage()` | `pages/components/DataPage.ets` | `cells`, `selectedDate`, `todayString`, `longestMonthStreak`, `monthCommitCount`, `onCellTap` |
| `logsPage()` | `pages/components/LogsPage.ets` | `checkIns`, `totalCommitCount` |
| `corePage()` | `pages/components/CorePage.ets` | `usagePatterns` |
| `userPage()` | `pages/components/UserPage.ets` | `themeName`, `totalCommitCount`, `usagePatterns.length`, `onSwitchTheme` |

Each component:
- Uses `@Consume tokens: DesignTokens` for theme
- Uses `@Consume themeName: SkinTheme` where needed (UserPage)
- Receives data via `@Prop` from parent
- Emits intents via callback props (`onCellTap`, `onSwitchTheme`)
- Is a Humble View — no business logic

### Step 4: Fix OCP violation in activePage()

Replace the `if/else` chain in `activePage()` with a `Map<AppSection, @Builder>` registry pattern. In ArkTS, we can use a `@Builder` method map or a simple switch in a thin dispatcher since ArkTS doesn't support `@Builder` in `Map` directly. The pragmatic approach: keep a thin switch in `HeatmapPage.activePage()` but each branch simply renders the corresponding component — this is acceptable because the page content is now fully encapsulated in components, and adding a new section = adding a new component + one line in the switch.

### Step 5: Remove dead code

- Delete `pages/HeatmapContent.ets` — not imported anywhere, superseded by the current `HeatmapPage.ets` structure
- Delete `pages/Index.ets` — boilerplate scaffold, not used (app loads `HeatmapPage`)

### Step 6: Clean up ThemeProvider.ets

`ThemeProvider.ets` is currently unused (HeatmapPage does its own `@Provide`). After Step 2, the theme persistence is in an adapter. Decide whether to:
- **Delete ThemeProvider.ets** (HeatmapPage is the entry point and handles `@Provide`)
- Or use it as a wrapper around HeatmapPage content

**Recommendation:** Delete it. HeatmapPage already provides tokens. ThemeProvider adds unnecessary indirection.

## Risks And Compatibility

- **ArkTS strict mode `Record<>` type**: `Record<SkinTheme, DesignTokens>` may need to become a `Map<SkinTheme, DesignTokens>` if ArkTS doesn't support `Record` with class values. Fallback: use a helper function `getThemePalette(theme: SkinTheme): DesignTokens` with a switch statement.
- **`HeatmapCell.checkIn` optional field**: Converting to a class means the `checkIn` field needs careful handling. In ArkTS, class fields can be optional with `checkIn?: CheckIn`, but need to verify this compiles.
- **`@Prop` reactivity with class types**: When `CheckIn` changes from interface to class, `@Prop` passing of `CheckIn[]` may behave differently. Verify that `ForEach` key generation still works.
- **Breaking test mocks**: Tests using object literals for `CheckIn`, `EasterEgg` will break. Need to update to `new CheckIn(...)`, etc.
