# Plan: 撰写 ARCHITECTURE-REVIEW.md

## Goal
在项目根目录创建 `ARCHITECTURE-REVIEW.md`，对当前代码库进行系统性架构审查，按严重程度分类列出所有发现的问题，附带具体文件位置和修复建议。

## Scope
- 审查范围：整个 `entry/src/main/ets/` + `entry/src/test/`
- 输出：单文件 `ARCHITECTURE-REVIEW.md`
- 不包含：代码修改（纯文档输出）

## 设计

经过完整代码阅读，已发现以下问题（按严重程度）：

### 🔴 严重（违反核心架构规则）

1. **Page 层直接使用 `@kit.ArkData`（preferences）** — `HeatmapPage.ets` 的 `loadPersistedTheme()` / `persistTheme()` 直接调用 `preferences.getPreferences()`，绕过了 adapter 层
2. **ThemeProvider 同样直接使用 `@kit.ArkData`** — `ThemeProvider.ets` 完全复制了主题持久化逻辑，且与 HeatmapPage 重复
3. **主题持久化逻辑重复 3 处** — HeatmapPage、ThemeProvider、SkinProviderImpl 各自管理主题持久化，且 HeatmapPage/ThemeProvider 未走 SkinProvider port
4. **DebugRuntime 直接构造 EasterEgg 对象字面量** — `DebugRuntime.egg()` 返回 object literal，违反 ArkTS 严格模式 + domain 实体应通过构造函数创建
5. **HeatmapVM `@Observed` 但通过 `AppStorage` 传递** — `@Observed` 用于 `@ObjectLink` 场景，但当前通过 `@StorageLink` 访问 VM，`@Observed` 装饰器无实际效果且可能引起混淆

### 🟡 中等（设计偏差/潜在风险）

6. **HeatmapContent.ets 是孤岛组件** — 定义了但未被任何地方引用，与 HeatmapPage 职能重叠
7. **Index.ets 是空白残留** — 未使用（路由指向 HeatmapPage），保留 Hello World 模板代码
8. **FrequencyPatternEngine 放在 domain 层但实现了 port** — domain 层应为纯实体，实现类应归入 usecases 或独立位置
9. **LearnedPatternPolicy 放在 domain 层但依赖无 port** — 策略类被 usecase 直接实例化，绕过了依赖注入
10. **LoadHeatmap 对 level 只设 0/1** — 未按 Streak.ets 定义的 0-4 级别做分级计算
11. **StreakCalculator port 存在但未被任何代码使用** — 定义了接口但无实现、无调用
12. **SoundPlayerImpl 未处理 `eva_alert` 频率** — `getFrequency()` 缺少 `eva_alert` 映射，回落到默认 660Hz
13. **ParticleOverlay 使用硬编码颜色** — 不从 DesignTokens 获取，与设计系统规范不一致
14. **DayCell/HeatmapGrid 中的 CELL_SIZE 硬编码** — 40vp 重复定义在两个文件，未使用 SpacingTokens
15. **HeatmapVM.doCheckIn() 中 patternLearning.match() 与 DetectEasterEgg usecase 重复** — 两套彩蛋检测逻辑并存

### 🟢 建议（改善可维护性）

16. **sectionLabel Builder 在 3 个 Section 中重复** — CalendarSection/LogsSection/PatternsSection/ProfileSection 各自定义了相同结构的 sectionLabel/summaryRow
17. **TypographyStyles.ets 定义了 @Extend 样式但从未使用** — h1Style/h2Style/bodyStyle/captionStyle 全部未被任何组件引用
18. **GlassTheme.ets 仅被极少数地方潜在使用** — 毛玻璃工具类存在但未见实际使用
19. **VMFixture 直接依赖 SkinProviderImpl（具体类）** — 测试 fixture 应使用 mock 而非真实 adapter
20. **EasterEgg interface 只有 keyword/quote/effectType/soundFile 4 个字段** — 作为 interface 可考虑改为 class 以符合 ArkTS 规范
21. **CheckIn interface 可改为 class** — 同理，ArkTS 更倾向 class
